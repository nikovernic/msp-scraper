# Story 6.2: Claim Verification Page & Profile Claiming API

## Status
Done

## Story
**As a** crew member,
**I want** to verify my claim token and claim my profile by creating an account,
**so that** I can manage my own profile information.

## Acceptance Criteria
1. Claim verification page displays at `/claim/[token]` route
2. Page validates claim token (checks existence, expiration, and that profile is not already claimed)
3. Page shows profile information (name, role, location) for verification
4. Page displays claim form with email and password fields
5. Form validation prevents submission with invalid email or weak password
6. Profile claiming API endpoint (POST /api/auth/claim) accepts token, email, and password
7. API creates Supabase Auth user account with provided email and password
8. API links profile to user account (sets user_id and is_claimed = true)
9. API invalidates claim token (sets claim_token and claim_token_expires_at to null)
10. API returns success response and redirects to profile edit page
11. API handles errors: invalid token (404), expired token (400), already claimed (400), email already exists (400)
12. All code follows coding standards and project structure

### Data Models
[Source: architecture/data-models.md#Profile, User]

Profile interface includes:
- `id`: UUID - Primary key
- `user_id`: UUID | null - Foreign key to User, null if unclaimed
- `is_claimed`: boolean - Whether profile has been claimed
- `claim_token`: string | null - Unique token for profile claiming
- `claim_token_expires_at`: string | null - Token expiration timestamp
- `name`: string - Crew member name (for display on verification page)
- `primary_role`: string - Role (for display)
- `primary_location_city`: string - City (for display)
- `primary_location_state`: string - State (for display)

User interface (Supabase Auth):
- `id`: UUID - Primary key, Supabase-generated
- `email`: string - Unique email address for login
- `role`: 'admin' | 'crew' - User role (default 'crew' for claimed profiles)

### Database Schema
[Source: architecture/database-schema.md#profiles, users]

Profiles table includes:
```sql
user_id UUID REFERENCES public.users(id) ON DELETE SET NULL,
is_claimed BOOLEAN DEFAULT FALSE,
claim_token TEXT UNIQUE,
claim_token_expires_at TIMESTAMPTZ,
```

Users table (Supabase Auth):
```sql
id UUID PRIMARY KEY,
email TEXT UNIQUE NOT NULL,
role TEXT NOT NULL CHECK (role IN ('admin', 'crew')) DEFAULT 'crew',
```

### API Specifications
[Source: architecture/api-specification.md#/auth/claim/{token}]

GET `/api/auth/claim/{token}` endpoint:
- Path parameter: `token` (string) - Claim token
- Responses:
  - 200: Token valid, returns profile information
  - 404: Token invalid or expired

POST `/api/auth/claim` endpoint:
- Request body:
  ```typescript
  {
    token: string;      // Required
    email: string;      // Required, email format
    password: string;   // Required, min 8 chars, complexity requirements
  }
  ```
- Responses:
  - 200: Profile claimed successfully
  - 400: Invalid request (validation error, expired token, already claimed, email exists)
  - 404: Token invalid
  - 500: Server error
