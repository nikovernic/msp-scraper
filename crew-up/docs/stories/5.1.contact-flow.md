# Story 5.1: Contact Flow

## Status
Done

## Story
**As a** producer,
**I want** to contact crew members through a contact form on their profile pages,
**so that** I can inquire about their availability and discuss potential projects.

## Acceptance Criteria
1. Contact form is displayed on individual profile pages (/crew/[slug])
2. Contact form includes required fields: producer name, email, message (max 500 chars)
3. Contact form includes optional fields: producer phone, shoot dates
4. Form validation prevents submission with invalid email or empty required fields
5. Contact form API endpoint (POST /api/profiles/[id]/contact) accepts and validates contact inquiries
6. Contact inquiries are stored in the contact_inquiries table with profile_id reference
7. Email notification is sent to the crew member's contact_email when a contact form is submitted
8. Email notification uses Resend API with React Email template
9. Rate limiting is implemented: maximum 5 contact form submissions per hour per IP address
10. Form submission shows success confirmation message to the producer
11. Form submission shows error message on validation or API errors
12. All code follows coding standards and project structure

### Data Models
[Source: architecture/data-models.md#ContactInquiry]

ContactInquiry interface includes:
- `id`: UUID - Primary key
- `profile_id`: UUID - Foreign key to Profile
- `producer_name`: string - Name of producer submitting inquiry
- `producer_email`: string - Email of producer
- `producer_phone`: string | null - Phone number (optional)
- `message`: string (max 500 chars) - Project details message
- `shoot_dates`: string | null - Shoot date information (optional)
- `created_at`: timestamp - Inquiry submission time

Profile interface (for email notification):
- `id`: UUID
- `name`: string - Crew member name (for email template)
- `contact_email`: string - Email address to send notification to
- `slug`: string - For generating profile link in email

### Database Schema
[Source: architecture/database-schema.md#contact_inquiries]

Contact inquiries table already exists:
```sql
CREATE TABLE public.contact_inquiries (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    profile_id UUID NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
    producer_name TEXT NOT NULL,
    producer_email TEXT NOT NULL,
    producer_phone TEXT,
    message TEXT NOT NULL CHECK (char_length(message) <= 500),
    shoot_dates TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

RLS policies:
- Anyone can create inquiries: `CREATE POLICY "Anyone can create inquiries" ON public.contact_inquiries FOR INSERT WITH CHECK (true);`
- Admins can view inquiries: `CREATE POLICY "Admins can view inquiries" ON public.contact_inquiries FOR SELECT USING (EXISTS (SELECT 1 FROM public.users WHERE id = auth.uid() AND role = 'admin'));`

Indexes available:
- `idx_contact_inquiries_profile` on (profile_id)
- `idx_contact_inquiries_created` on (created_at)

### API Specifications
[Source: architecture/api-specification.md#/profiles/{id}/contact]

POST `/api/profiles/{id}/contact` endpoint:
- Path parameter: `id` (UUID) - Profile ID
- Request body:
  ```typescript
  {
    producer_name: string;      // Required
    producer_email: string;     // Required, email format
    producer_phone?: string;    // Optional
    message: string;            // Required, max 500 chars
    shoot_dates?: string;       // Optional
  }
  ```
- Responses:
  - 200: Inquiry submitted successfully
  - 400: Invalid request (validation error)
  - 429: Rate limit exceeded
  - 404: Profile not found
  - 500: Server error

### Rate Limiting Implementation
[Source: architecture/security-and-performance.md#Rate Limiting]

Rate limiting requirements:
- Contact forms: 5 per hour per IP address
- Use Vercel Edge Config or similar for rate limiting
- Extract client IP from request headers (`x-forwarded-for` or `x-real-ip`)
- Return 429 (Too Many Requests) status code when rate limit exceeded
- Rate limit check should happen before processing submission

Note: For MVP, a simple in-memory rate limiting solution may be acceptable for local development, but production should use Vercel Edge Config or Redis for distributed rate limiting.
